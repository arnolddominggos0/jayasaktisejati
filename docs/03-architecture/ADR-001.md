## ADR-001

### Title
Single Laravel Application

### Status
Accepted

### Context
The Jaya Sakti Line Website MVP is a public marketing and lead-generation website (PRD-001). The project brief mandates that the website **will NOT become a standalone application**; it must become part of the existing JayaSaktiApp ecosystem — the existing `jss_dashboard` Laravel 11 repository, which already hosts an operational logistics system with three Filament panels, Sanctum, Spatie Permission, and Spatie Activity Log.

The Architecture Review (AR-01) evaluated three topology options: a monolith inside the existing app, a same-codebase separate process, and a separate app sharing the database. The ARB accepted the monolith-inside-existing-app option.

### Decision Drivers
1. Project brief: the website is not a standalone application.
2. Stack is fixed to Laravel 11 / PHP 8.3 / Filament v3 / MariaDB.
3. MVP traffic is low (PRD KPI: ≥ 500 vessel listing views/month).
4. Need to maximize reuse of existing auth, media, config, and Filament infrastructure.
5. Minimize operational cost and time-to-MVP.

### Decision
The Jaya Sakti Line website runs **inside the existing `jss_dashboard` Laravel application** as a single, shared runtime process. There is no second application, no separate codebase, and no separate deployment artifact for the website.

### Consequences
- One codebase, one deployment pipeline, one runtime.
- Public traffic shares server resources with the operational admin system.
- Website code is added as new modules/namespaces within the existing app, not as a separate project.
- A public traffic spike could, in principle, affect operational admin performance — mitigated by caching (deferred to ARD-001) and hosting/scaling choices, not by splitting the codebase.

### Benefits
- Maximum reuse of existing infrastructure (auth, media, config, Filament, queues, mail).
- Lowest MVP delivery cost and fastest time-to-market.
- Single deploy and single source of operational truth.
- Consistent with the explicit project constraint that the website is not standalone.

### Drawbacks
- Public and operational traffic share the same process (no runtime isolation).
- A single deploy ships both website and operational changes (shared blast radius).
- Public-facing bugs theoretically share an attack surface with internal admin.

### Future Considerations
- If public traffic grows substantially, resource isolation can be achieved via caching, a CDN, or a separate worker pool **without splitting the codebase** — these are operational/infrastructure concerns, not application-architecture concerns.
- Any move to a separate runtime would require a new ADR superseding ADR-001 and a re-evaluation against the then-current traffic profile.

## ADR-002

### Title
Modular Monolith Architecture

### Status
Accepted

### Context
The website and the operational logistics system coexist in one Laravel app (ADR-001). PRD-001 §8.2 explicitly excludes operational features (voyage, shipment, KPI, CRM, AIS) from the MVP, and Risk R5 (scope creep) is rated High. Without a hard internal boundary, operational code and marketing code would intermingle, making scope creep inevitable and future multi-business expansion (PRD Phase 5) expensive.

The Architecture Review (AR-02, AR-03) recommended a module/namespace partition with a "no operational imports" rule, structured under a business-unit-scoped module from day one.

### Decision Drivers
1. PRD scope boundary: operational features are out of scope (PRD §8.2).
2. Risk R5: scope creep into operational features is High impact.
3. PRD Phase 5 [FUTURE]: other Jaya Sakti business units may join the platform.
4. Need enforceable separation that is CI-checkable, not just convention.
5. Architecture style is fixed as Modular Monolith.

### Decision
The application is organized as a **Modular Monolith**. The Jaya Sakti Line website is implemented as a dedicated, business-unit-scoped module with its own namespace and folder root. A strict dependency rule applies: **the website module must not import or depend on operational namespaces, services, models, or Filament resources.** This rule is enforceable in code review and CI.

A "platform kernel" (shared infrastructure: auth, media primitives, configuration, routing conventions) is distinguished from the "Line unit" (the website's own controllers, models, Filament resources, views). Only the platform kernel is shared; the Line unit owns its own domain code.

### Consequences
- The website module is independently understandable and testable.
- Operational and marketing domains do not share domain models.
- Adding a second business unit in the future is an additive change (a new sibling module), not a restructure.
- A CI rule/static review must be maintained to prevent operational imports from the website module.

### Benefits
- Enforceable scope boundary; directly mitigates Risk R5.
- Clean ownership and clear module responsibilities.
- Cheapest path to PRD Phase 5 multi-business readiness (see ADR-010).
- No distributed-system complexity (no network calls between modules).

### Drawbacks
- Discipline required to maintain the dependency rule.
- Some minor duplication of generic concepts if the platform kernel is kept intentionally thin.
- A "platform kernel vs Line unit" classification must be documented in ARD-001 to avoid ambiguity.

### Future Considerations
- If a second business unit is approved, a shared platform kernel is formalized and per-unit modules are added as siblings.
- If a module ever needs to become a separate service, the module boundary is the natural extraction seam (modular monolith → extracted service), but this is far out of scope for MVP.

## ADR-003

### Title
Dedicated CMS Panel

### Status
Accepted

### Context
The existing `jss_dashboard` app already has three Filament panels: `admin` (at `/admin`), `field-coordinator`, and `customer`. The operational `admin` panel is configured with `EnsurePanelRole` and `ScopeByBranch` middleware — its data is branch-scoped and operational. The PRD-001 CMS, by contrast, manages single-tenant marketing content (company profile, services, vessel listings, gallery, inquiries) that has no branch dimension and a single admin role (PRD FR-07).

Folding CMS resources into the operational `admin` panel would drag branch-scoping middleware onto non-branch content, clutter operational navigation, and erode the boundary established in ADR-002. The Architecture Review (AR-05) recommended a dedicated new Filament panel.

### Decision Drivers
1. Operational `admin` panel carries `ScopeByBranch` — irrelevant and harmful for marketing content.
2. PRD FR-07: single admin role for CMS; additional roles are [FUTURE].
3. Need to preserve the module boundary (ADR-002).
4. Desire to reuse Filament's CRUD/upload/table strengths without operational coupling.
5. Independent navigation and auth gating for the CMS.

### Decision
A **dedicated, fourth Filament panel** is created for the CMS, with its own path, its own navigation, and its own middleware stack that **excludes `ScopeByBranch`**. The CMS panel uses the existing `web` guard and users table but gates access via a dedicated CMS role (see ADR-008 and the auth boundary). The CMS panel is the only admin surface for website content; it does not share resources or navigation with the operational `admin` panel.

### Consequences
- Four Filament panels coexist in the app: operational `admin`, `field-coordinator`, `customer`, and the new CMS panel.
- The CMS panel's middleware stack is independent of operational branch scoping.
- CMS navigation is isolated and marketing-focused.
- A user who needs both operational and CMS access must be granted the respective role for each panel and switch between them.

### Benefits
- Clean boundary: marketing content is never branch-scoped.
- No operational navigation clutter; each panel is purpose-built.
- Reuses Filament's full CRUD/upload/table UX for the CMS at near-zero infrastructure cost.
- Independent auth/role gate for the CMS, supporting the single-admin MVP and a clean path to future roles.

### Drawbacks
- One additional panel to register and maintain.
- A user with both operational and CMS roles experiences panel switching.

### Future Considerations
- Future CMS roles (editor, broker-only — PRD §19 Phase 2/3) are additive to this panel via Spatie Permission, not a new panel.
- The panel's role gate (extending the existing `EnsurePanelRole` pattern) is detailed in ARD-001.

## ADR-004

### Title
Separate Marketing Domain Model — Do Not Reuse the Operational Vessel Model

### Status
Accepted

### Context
The existing `App\Models\Vessel` is an operational domain model used by the logistics system (voyage, vessel check, vessel plan). PRD-001 §15 defines a fundamentally different concept: a marketing "vessel listing" — an opportunity posted by Jaya Sakti Line as broker, with general public information, a Open/Closed lifecycle, up to six images, and sensitive internal fields (real vessel name, IMO, owner, certificates) that must never be public (PRD AC-4).

Reusing the operational `Vessel` would leak operational fields into public pages, fuse two bounded contexts with different lifecycles and audiences, and make AC-4 (no sensitive data on public surfaces) effectively unenforceable. The Architecture Review (AR-06) recommended a distinct marketing entity with no relationship to the operational `Vessel`.

### Decision Drivers
1. PRD AC-4: sensitive data must never appear in public HTML, responses, images, or alt text.
2. PRD §15: the marketing listing is a different concept from an operational vessel.
3. ADR-002: the website module must not depend on operational domain code.
4. Different lifecycles, fields, and audiences between the two concepts.
5. Risk R1 (sensitive data leakage) is High impact.

### Decision
The website module defines its **own marketing domain entity** for vessel opportunities (a `VesselListing` concept), located within the website module's namespace. This entity has **no relationship, foreign key, or code dependency** on the operational `App\Models\Vessel`. The two are independent bounded contexts that happen to share the word "vessel."

The marketing entity owns its own general-information fields, its own sensitive internal fields, its own image associations, and its own Open/Closed status. Sensitive fields are handled per ADR-005.

### Consequences
- Two vessel-related concepts exist in the codebase (operational `Vessel` and marketing `VesselListing`). This is intentional and correct.
- The website module's dependency rule (ADR-002) is reinforced: no import of operational models.
- Public data access is structurally decoupled from operational data.
- Any future link between a marketing listing and a real operational vessel is explicitly a [FUTURE] integration (PRD Phase 6), not an MVP concern.

### Benefits
- AC-4 becomes enforceable at the data-access layer — public code paths simply never touch operational vessel data.
- Clean bounded contexts with independent evolution.
- No risk of operational field leakage into public surfaces.
- The marketing entity's schema can be tailored to PRD §15 without affecting operational tables.

### Drawbacks
- Conceptual duplication of the word "vessel" requires clear naming to avoid developer confusion.
- Stakeholders may later want a marketing listing to reference a real operational vessel — this is intentionally deferred.

### Future Considerations
- If PRD Phase 6 (operational integration) is approved via Change Request, a loose reference (e.g., an optional nullable external identifier) can be added **without** merging the models. That integration requires a new ADR.
- Naming conventions for the marketing entity and its tables are specified in the Database Design (DBD-001), not here.

## ADR-005

### Title
Public Projection Pattern for Sensitive Vessel Data

### Status
Accepted

### Context
PRD-001 §15.2 and AC-4 require that sensitive vessel fields — real vessel name, IMO number, owner/ownership details, full certificates, and (recommended) price/commercial terms — must **never** appear in any public page, API response, HTML source, alt text, or downloadable asset. Risk R1 (sensitive data leakage) is High impact. PRD §20.4 mandates a "sensitive data leakage test" as a hard launch gate.

Single-layer filtering (a single transformer that strips sensitive fields before render) was considered but rejected as a single point of failure: one missed endpoint or template leaks commercially sensitive data. The Architecture Review (AR-12) recommended defense in depth.

### Decision Drivers
1. PRD AC-4 is a hard, non-negotiable acceptance criterion.
2. Risk R1 is High impact; sensitive data is commercially and legally sensitive.
3. PRD §20.4 hard gate requires an automated leakage test before launch.
4. Need assurance that survives developer error and future endpoint additions.

### Decision
Sensitive vessel data is protected by a **Public Projection Pattern** with defense in depth:

1. **Public projection** — public-facing code paths consume a dedicated read shape ("public projection") that **structurally excludes** sensitive columns. Public controllers, views, and API responses are fed only this projection; they do not receive the full entity. Sensitive fields are not merely hidden in templates — they are not selected/loaded into the public shape at all.
2. **No direct model-to-view passing** for vessel listings in public contexts. The full entity (with sensitive fields) is only available within the CMS/admin context.
3. **Image EXIF/metadata stripping on upload** — vessel and gallery images are stripped of EXIF/metadata that could leak owner or vessel identity (PRD §15.4).
4. **Automated leakage test** — a parameterized test asserts the absence of sensitive fields across all public routes/responses (per ADR's testing strategy; formalized in ARD-001 and DBD-001).
5. **Allow-list, never block-list** — for any future public field exposure, only an explicit allow-list is permitted.

### Consequences
- Public endpoints are structurally incapable of selecting sensitive columns.
- The full entity is loaded only inside the CMS panel.
- Image uploads are processed (EXIF stripped) before storage.
- An automated test guards against regressions across all public routes.
- A developer cannot "accidentally" render a sensitive field without bypassing the projection — which becomes a visible, reviewable act.

### Benefits
- Multiple independent controls; a failure in one layer does not leak data.
- AC-4 assurance is structural, not convention-based.
- Directly satisfies the PRD §20.4 hard gate.
- The leakage test is parameterized over all public routes, so new endpoints are covered automatically.

### Drawbacks
- Slightly more structure (a projection type/shape per public-facing entity).
- Discipline required to forbid direct model-to-view passing in public contexts.

### Future Considerations
- If PRD Phase 6 introduces an operational integration, any data crossing into public surfaces must pass through the same public projection — never bypass it.
- The exact shape of the projection (DTO/resource) is specified in ARD-001 and DBD-001.

## ADR-006

### Title
Single Database Strategy

### Status
Accepted

### Context
ADR-001 places the website inside the existing `jss_dashboard` Laravel application. The existing app uses a single MariaDB database for the operational system. ADR-004 separates the marketing domain model from the operational `Vessel`, but that is a *logical* separation, not a storage separation.

The question is whether the website module's tables live in the same MariaDB database as the operational tables, or in a separate database. The Architecture Review and the modular-monolith style (ADR-002) favor a single database with logical separation by module, since the website shares no operational *data* in MVP and a second database would add cross-database operational complexity for no benefit.

### Decision Drivers
1. ADR-001: single application runtime.
2. ADR-002: modular monolith with module-level separation.
3. ADR-004: marketing domain is logically separate but lives in the same app.
4. Stack is fixed to MariaDB.
5. No operational data is shared with the website in MVP.

### Decision
The website module stores its data in the **same single MariaDB database** as the operational system. Separation between marketing and operational domains is enforced **logically** — via module namespaces, table naming conventions, and the dependency rule (ADR-002) — not via a separate database.

The website module's tables are namespaced/prefixed to distinguish them from operational tables and to keep the module self-contained. No foreign keys cross the marketing/operational boundary in MVP.

### Consequences
- One database to back up, monitor, and migrate.
- Marketing and operational tables coexist but are logically isolated by naming and module ownership.
- No cross-database joins or distributed transactions.
- Migrations for the website module are kept within the website module's ownership.

### Benefits
- Simplest operational model; one backup, one restore, one connection.
- Consistent with modular-monolith style (logical separation, single store).
- No distributed-data complexity.
- Future extraction to a separate database is still possible if scale demands, because the logical boundary is already in place.

### Drawbacks
- Shared database means a migration or query incident could theoretically affect both domains.
- Table namespace/prefix convention must be maintained.

### Future Considerations
- If a business unit is added (ADR-010) and data volume grows, a per-unit database split can be reconsidered via a new ADR; the logical naming boundary makes that split mechanical.
- Table prefix/naming conventions are defined in DBD-001.

## ADR-007

### Title
Public Website and CMS in the Same Application

### Status
Accepted

### Context
PRD-001 defines two surfaces: a public website (company profile, vessel trading, gallery, inquiry) and a CMS (admin content management). PRD §20.1 recommends keeping "public site and CMS as one Laravel app with separate route groups." The Architecture Review (AR-01, AR-07) confirmed that both surfaces live in the single application (ADR-001).

The architectural question is whether the public website and the CMS are one app with separated route groups, or whether the CMS is a separate app. Given ADR-001, they are one app; this ADR formalizes how the two surfaces are kept apart within that one app.

### Decision Drivers
1. ADR-001: single application.
2. PRD §20.1: one Laravel app with separate route groups.
3. Need clear separation between anonymous public routes and authenticated admin routes.
4. Avoid accidental exposure of admin routes on the public surface.

### Decision
The public website and the CMS are **two surfaces of the same application**, separated by route groups and middleware:

- **Public surface** — anonymous, SEO-friendly routes at the root (`/`, `/about`, `/services`, `/vessels`, `/vessels/{ref}`, `/gallery`, `/contact`). Rendered server-side (Blade). No authentication. Protected by CSRF, spam protection, and the public projection (ADR-005).
- **CMS surface** — authenticated routes under the dedicated CMS panel path (ADR-003), gated by the CMS role, served by Filament (ADR-008). Not indexed, not linkable from public navigation.

The two surfaces share the same codebase, database (ADR-006), and platform kernel, but they have distinct route files, distinct middleware stacks, distinct views, and distinct authorization. There is no public route that exposes CMS functionality and no CMS route that serves public marketing pages.

### Consequences
- One application, two clearly separated surfaces.
- Public routes are anonymous and cached (per ARD-001); CMS routes are authenticated and uncached.
- A single deploy updates both surfaces simultaneously.
- Public and CMS UIs are visually and structurally distinct (separate components per ADR-002/AR-09).

### Benefits
- Simplest delivery and operations (one app, one deploy).
- Shared platform kernel (auth, media, config) reused across surfaces.
- Clear security boundary: admin functionality is never reachable from public routes.

### Drawbacks
- A single deploy changes both surfaces at once (shared blast radius) — mitigated by feature flags and the module boundary.
- Public and CMS performance characteristics differ and must be tuned independently (caching for public, none for CMS).

### Future Considerations
- If the CMS ever needs to be exposed as a headless API for a separate marketing front-end, the route-group boundary is the natural API extraction seam — but this is out of MVP scope.
- Routing conventions (route-file organization, route-name prefixes) are specified in ARD-001.

## ADR-008

### Title
Filament as the Internal CMS Framework

### Status
Accepted

### Context
PRD-001 requires a CMS with CRUD for company profile, services, vessel listings (with image upload and ordering), gallery, and an inquiry inbox. The existing `jss_dashboard` app already uses Filament v3 for its operational panels and has Filament installed and configured. Building a custom non-Filament admin area would re-implement CRUD, file upload, tables, forms, and filtering that Filament already provides. The Architecture Review (AR-05, AR-11) recommended Filament for the CMS, with Spatie Permission (already present) for authorization.

### Decision Drivers
1. Filament v3 is already installed and proven in the existing app.
2. PRD FR-08/FR-09 require rich CRUD, image upload with ordering, and an inquiry inbox — all native Filament capabilities.
3. PRD FR-07: single admin role for MVP; future roles are additive.
4. Spatie Permission is already present for role-based access.
5. Minimize custom UI infrastructure for the CMS.

### Decision
The CMS is built **on Filament v3**, as a dedicated panel (ADR-003). All CMS management surfaces — company profile editor, services manager, vessel listings manager (with ≤6 image upload + ordering + Open/Closed toggle), gallery manager, and inquiry inbox — are implemented as Filament resources within the website module's namespace.

Authorization uses **Spatie Laravel Permission** with a single CMS role for MVP and resource-level policies (Eloquent policies) for every CMS resource. This makes future roles (editor, broker-only — PRD Phase 2/3) an additive change rather than a restructure. Inline authorization checks (e.g., ad-hoc `if (admin)`) are prohibited; all authorization goes through policies.

The CMS panel uses the existing `web` guard and the existing users table, gated by the dedicated CMS role, and **excludes operational branch-scoping middleware** (per ADR-003).

### Consequences
- CMS UX is consistent with the rest of the JayaSaktiApp admin ecosystem (Filament).
- CMS resources are namespaced within the website module (ADR-002), not mixed with operational Filament resources.
- Authorization is policy-based and role-driven from day one.
- Adding future CMS roles is a permission/role change, not an architecture change.

### Benefits
- Reuses proven Filament CRUD/upload/table/filter UX at near-zero infrastructure cost.
- Consistent admin experience across the ecosystem.
- Future-proof authorization via Spatie Permission + policies.
- No custom admin framework to maintain.

### Drawbacks
- CMS UI is constrained to Filament's design system (acceptable for an internal tool).
- A fourth Filament panel is registered (per ADR-003).

### Future Considerations
- Future broker accounts and inquiry assignment (PRD Phase 2) are added as new roles/permissions on this panel.
- Any CMS resource that must not be branch-scoped must never inherit operational middleware — enforced by the panel's middleware stack (ADR-003).

## ADR-009

### Title
Media Storage Strategy

### Status
Accepted

### Context
PRD-001 §15.4 requires up to 6 images per vessel listing with admin-controlled ordering, first image as thumbnail, and EXIF/metadata stripping to prevent owner/identity leakage. PRD §14.5 recommends auto-resize to web-optimized dimensions, max image size, and obfuscated public paths. PRD NFRs require LCP < 3s on 4G mobile (AC-8) and lazy-loaded, responsive images (Risk R6). The existing app configures `local`, `public` (with `storage:link`), and `s3` filesystem disks.

The Architecture Review (AR-13) recommended local disk for MVP with responsive variants generated on upload and EXIF stripping, with the media abstraction written so S3/CDN is a configuration swap later.

### Decision Drivers
1. PRD AC-8: LCP < 3s on 4G mobile.
2. PRD AC-4 / §15.4: image EXIF must not leak owner/vessel identity.
3. PRD §14.5: auto-resize, obfuscated public paths, private admin storage.
4. MVP traffic and volume are low; operational cost must stay low.
5. Future scalability to S3/CDN should not require a rewrite.

### Decision
Media for the website module is stored on the **local `public` disk** (with `storage:link`) for MVP, under **obfuscated paths** (non-guessable filenames) within a website-module-owned directory. The media pipeline performs, on upload:

1. **EXIF/metadata stripping** — all uploaded images are stripped of EXIF/metadata before storage (supports ADR-005 and AC-4).
2. **Responsive variant generation** — web-optimized resized variants are generated on upload (e.g., thumbnail, listing, detail) to support responsive `srcset` and lazy loading on the public site.
3. **Size limit** — a maximum upload size is enforced (per PRD §14.5).
4. **Obfuscated filenames** — public-facing media uses non-guessable filenames; admin-private sensitive media (if any) is stored on a non-public disk.

The media handling is implemented behind a **media abstraction** (a media service/port within the website module) so that swapping to S3 or a CDN in the future is a configuration change, not a code rewrite. Admin-only media (e.g., sensitive internal references) is never placed on the public disk.

### Consequences
- Public media is served from local disk via `storage:link`, with responsive variants and EXIF stripped.
- Image HTML on the public site uses responsive `srcset` + lazy loading to meet AC-8.
- The media abstraction localizes all disk/format concerns in one place.
- A future move to S3/CDN is a disk-configuration change behind the abstraction.

### Benefits
- Meets AC-8 (responsive, lazy-loaded images) and AC-4 (EXIF stripping) at MVP cost.
- No external storage cost or dependency for MVP.
- Obfuscated paths reduce predictable media enumeration.
- S3/CDN path preserved without building it now.

### Drawbacks
- Local disk scaling limits (acceptable for MVP volume).
- Local disk media must be included in the backup strategy (PRD NFR: weekly DB + media backup).
- Server migration requires migrating local media.

### Future Considerations
- When traffic or volume grows, switch the media abstraction's disk to S3/CDN via configuration — no public-site code change.
- Exact variant sizes, format (e.g., WebP), and naming are specified in ARD-001.
- Backup inclusion of media is formalized in the operational/backup section of ARD-001.

## ADR-010

### Title
Future Multi-Business Ready Architecture

### Status
Accepted

### Context
PRD-001 BO-6 [RECOMMENDATION] and Phase 5 [FUTURE] anticipate that other Jaya Sakti business units may join the platform beyond Jaya Sakti Line. Risk R7 warns that architecture which ignores this future could force a re-platforming later. However, PRD scope is frozen and "no future scope" must be built in the MVP. The Architecture Review (AR-19, AR-20) recommended "structural readiness without speculative features" — build one unit now, but structure the architecture so a second unit is additive, not a rewrite.

### Decision Drivers
1. PRD Phase 5 [FUTURE]: multi-business platform.
2. Risk R7: avoid re-platforming later.
3. "No future scope" constraint — do not build speculative features.
4. ADR-002: modular monolith with a platform-kernel vs unit distinction.

### Decision
The architecture is **structurally ready** for multiple business units, without building any multi-tenant feature in the MVP:

1. **Business-unit-scoped module** — the Jaya Sakti Line website lives under a "Line" unit namespace from day one (ADR-002/ADR-003). A second unit would be added as a sibling module, not a restructure.
2. **Platform kernel vs unit ownership** — shared infrastructure (auth, media abstraction, routing conventions, configuration, navigation primitives) is owned by the platform kernel; unit-specific code (controllers, models, Filament resources, views, brand) is owned by the unit. ARD-001 defines exactly which concerns belong to each.
3. **No hardcoded "Line" assumptions in the kernel** — the platform kernel does not reference Line-specific domain concepts; brand/domain specifics live in the unit.
4. **Documented extension points** — explicit, named seams are left for future phases (inquiry pipeline → CRM sync; listing lifecycle → owner-submitted drafts; media abstraction → CDN; auth/roles → broker accounts; integration port → operational data). Each extension point is a one-line seam (interface/port or naming convention), **not** implemented code.
5. **No speculative multi-tenant substrate** — there is no tenant-selection mechanism, no per-unit admin switching, and no per-unit database routing in the MVP.

### Consequences
- The MVP contains one business unit (Jaya Sakti Line) but is structured so a second unit is additive.
- A future second unit requires a new module and brand config, not a rewrite of the kernel.
- Extension points are documented and named, steering MVP code away from dead-ends.
- No speculative abstraction is built; the MVP stays minimal.

### Benefits
- Cheapest possible insurance against Risk R7.
- Phase 5 (multi-business) becomes an additive evolution rather than a re-platform.
- Extension points make PRD §19 phases (2–6) implementable without rewiring the MVP.
- Complies with "no future scope" — readiness is structural, not feature-based.

### Drawbacks
- Requires ARD-001 to clearly classify "platform kernel" vs "Line unit" concerns; ambiguity could erode the benefit.
- Slight discipline cost to keep the kernel free of Line-specific assumptions.

### Future Considerations
- Each PRD Phase 2–6 item (CRM, owner portal, trust features, multi-business, operational integration) requires a Change Request against frozen PRD-001 and a corresponding new ADR before implementation.
- When a second business unit is approved, the platform kernel is formalized and the unit module is added as a sibling; this may warrant a new ADR superseding portions of this one.

## ADR Supersession Policy

These ADRs are **Accepted and immutable**. To change an Approved decision:

1. Open a new ADR (ADR-011, ADR-012, …) that explicitly **supersedes** the target ADR.
2. The superseded ADR's status changes to **Superseded by ADR-XXX**.
3. The new ADR is reviewed and approved by the Architecture Review Board.
4. Any change with product-scope impact additionally requires a Change Request against frozen PRD-001.
